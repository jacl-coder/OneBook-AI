# 大模型核心与 Agent 关键机制

> 用途：梳理大模型 Agent 应用研发的核心知识点，适合学习路线设计、项目拆解和面试准备。

## 1. 大模型核心

### 1.1 Prompt 设计

- 是什么：把任务目标、约束、输入格式、输出格式，用模型更容易执行的方式表达出来。
- 为什么重要：很多效果问题不是模型不行，而是指令不清、上下文混乱、输出要求不明确。
- 学到什么程度：
  - 会写 `system` / `user` / `tool` 角色分工
  - 会做 few-shot 示例
  - 会限制输出格式、边界、拒答条件
  - 会针对“提取、分类、总结、改写、推理”写不同 prompt
- 常见误区：把 prompt 当玄学。实际上它更像“接口契约设计”。

### 1.2 Function calling / tool calling

- 是什么：让模型不直接瞎编，而是决定何时调用外部工具，比如查数据库、搜网页、发邮件、执行代码。
- 为什么重要：Agent 真正有用，靠的不是“会聊天”，而是“会调用外部能力”。
- 学到什么程度：
  - 理解模型输出“调用哪个工具、传什么参数”
  - 会设计清晰的 tool schema
  - 会处理工具失败、超时、重试
  - 会做“模型决定调用”与“程序强制调用”的区分
- 核心点：模型负责决策，系统负责执行和兜底。

### 1.3 Structured output

- 是什么：要求模型输出固定 JSON、枚举值、表单字段，而不是自由文本。
- 为什么重要：工程里下游程序要消费结果，自由文本最不稳定。
- 学到什么程度：
  - 会定义 JSON Schema / Pydantic / Zod
  - 会校验字段类型、必填项、枚举值
  - 会做输出修复和失败重试
- 核心点：能结构化，就尽量不要让模型自由发挥。

### 1.4 上下文管理、记忆、会话状态

- 是什么：
  - 上下文管理：本轮给模型哪些信息
  - 记忆：哪些历史信息长期保留
  - 会话状态：当前任务进行到哪一步
- 为什么重要：模型本身不“记得”你，系统要自己管理历史和状态。
- 学到什么程度：
  - 会做短期记忆：最近几轮对话裁剪
  - 会做长期记忆：用户偏好、历史事实存储
  - 会区分“聊天记录”和“业务状态”
- 关键区别：
  - 聊天历史是文本上下文
  - 状态是程序可读的任务进度

### 1.5 RAG 基础：切分、embedding、向量检索、rerank

- 是什么：把外部知识接进模型，让它基于你的资料回答。
- 为什么重要：很多业务问题不是靠模型参数，而是靠外部知识库。
- 拆开看：
  - 切分：把文档拆成适合检索的小块
  - embedding：把文本转成向量，便于语义检索
  - 向量检索：从知识库找到语义最相关片段
  - rerank：对召回结果再次排序，提高相关性
- 学到什么程度：
  - 知道 chunk 太大/太小的影响
  - 会建立索引并做 top-k 检索
  - 会把检索结果塞进 prompt
  - 会评估召回质量，不只看生成结果
- 常见误区：RAG 不是“把文档扔进向量库就行”，难点主要在检索质量。

### 1.6 模型评估：准确率、幻觉、延迟、成本

- 是什么：衡量系统到底是否可上线。
- 为什么重要：Agent 应用不做评估，基本等于靠感觉开发。
- 核心指标：
  - 准确率：回答是否正确、工具调用是否正确
  - 幻觉：是否编造事实、引用不存在内容
  - 延迟：响应要多久，用户能不能接受
  - 成本：token、工具调用、推理链长度是否可控
- 学到什么程度：
  - 会建立测试集
  - 会区分离线评估和线上监控
  - 会记录每一步调用日志
  - 会比较不同 prompt / 模型 / 检索策略
- 核心点：没有 eval，就无法稳定优化。

## 2. Agent 关键机制

### 2.1 ReAct / Plan-and-Execute / 多 Agent 协作

#### ReAct

- 是什么：边推理边行动，想一步做一步。
- 优点：简单灵活，适合中短任务。
- 风险：容易绕圈、步骤不稳定。

#### Plan-and-Execute

- 是什么：先规划，再逐步执行。
- 优点：更适合长任务，结构清晰。
- 风险：计划可能一开始就错。

#### 多 Agent 协作

- 是什么：把不同角色拆给不同 agent，比如规划者、执行者、审查者。
- 优点：复杂任务可分工。
- 风险：成本高、调试难、收益常被高估。

- 建议：先学单 Agent + workflow，再碰多 Agent。

### 2.2 工具使用：搜索、数据库、代码执行、浏览器操作

- 是什么：让 Agent 连接真实世界能力。
- 典型工具：
  - 搜索：获取实时信息
  - 数据库：查业务数据
  - 代码执行：计算、数据处理、生成文件
  - 浏览器操作：登录、点击、抓页面内容
- 学到什么程度：
  - 会为每种工具定义输入输出
  - 会做权限限制
  - 会记录调用日志
  - 会在工具结果不可信时加校验
- 核心点：工具越强，风险越高，必须加约束。

### 2.3 工作流编排：任务拆解、重试、超时、回退

- 是什么：把 Agent 从“聊天”变成“稳定系统”。
- 关键能力：
  - 任务拆解：大任务分成多个可执行步骤
  - 重试：工具失败后自动再试
  - 超时：避免一直卡死
  - 回退：某步失败时用备用方案或人工接管
- 学到什么程度：
  - 会设计状态机或 DAG 流程
  - 会区分“可重试错误”和“不可重试错误”
  - 会设置最大重试次数和熔断
- 核心点：生产环境里，编排能力比 prompt 更重要。

### 2.4 长任务状态管理

- 是什么：任务执行几分钟甚至几小时，系统需要记住执行进度。
- 为什么重要：很多真实 Agent 任务不是一次请求就完成。
- 学到什么程度：
  - 会把任务状态存数据库/队列
  - 会支持暂停、恢复、取消
  - 会记录中间结果和失败原因
- 典型状态：`pending` / `running` / `waiting` / `failed` / `completed`

### 2.5 权限控制和安全沙箱

- 是什么：限制 Agent 能做什么、能访问什么、在哪个环境运行。
- 为什么重要：Agent 一旦有工具能力，就可能误删数据、越权访问、执行危险操作。
- 需要掌握：
  - 工具白名单
  - 参数校验
  - 用户身份与资源隔离
  - 敏感操作二次确认
  - 代码执行沙箱、文件系统隔离、网络隔离
  - 防 prompt injection
- 核心点：安全不是补丁，是 Agent 架构的一部分。

## 3. 一句话理解

- 大模型核心解决的是：模型怎么“说得对”。
- Agent 关键机制解决的是：系统怎么“做得成、做得稳、做得安全”。

## 4. 学习顺序建议

1. Prompt 设计
2. Structured output
3. Function calling
4. 上下文和会话状态
5. RAG
6. 单 Agent 工作流
7. 工具接入
8. 评估体系
9. 长任务和安全
10. 最后再学多 Agent
