# OneBook AI RAG 产品化差距报告

## 1. 结论

当前项目已经具备一个可用的 RAG 主链，能够完成：

- 文档上传、解析、分块、索引
- `dense + lexical + fusion + rerank` 混合检索
- 基于证据生成答案、返回引用、证据不足时拒答
- 会话持久化与基础离线评测

但如果以“产品级 RAG 系统”为标准来看，当前仍然更接近“可用版本”或“工程化原型”，距离成熟产品还有几类关键能力缺口。

一句话概括：

**现在已经有 RAG 主链，也已有初版路由与拒答护栏，但还缺少更强的记忆、上下文治理、线上评测与运维闭环。**

## 2. 当前已具备的基础

从现有代码与文档来看，项目已经完成的基础包括：

- 文档处理链路完整：上传、解析、OCR、分块、索引、状态流转
- 检索链路完整：Qdrant + OpenSearch 混合检索，支持 rerank
- 生成链路完整：证据拼装、引用返回、拒答处理
- 基础会话能力：消息持久化、会话续聊、最近历史拼接
- 管理与评测基础：后台管理、RAG Eval Center、离线评测脚本

这些能力说明项目已经跨过了“Demo 阶段”，但还没有形成稳定、可持续演进的产品闭环。

## 3. 距离产品级还缺哪些能力

### 3.1 查询路由能力

当前系统已经有一层初版 query route，能识别部分“只依赖历史”的跟进问题，以及部分明显书外的实时问题。

产品级系统通常需要先判断：

- 这个问题是否真的需要检索
- 是否只依赖当前会话历史
- 是否明显超出书籍知识边界
- 是否应该直接拒答或走其他数据源

当前问题不再是“完全没有路由”，而是这层路由还比较浅，主要是规则驱动，覆盖范围和可配置性都有限。继续补齐后，才能稳定支撑：

- 不该检索的问题也进入向量检索，带来延迟和噪声
- 用户问到书外问题时，系统很容易表现得不自然或不稳定

### 3.2 记忆体系不完整

当前有会话持久化，但还没有完整的 memory 分层。

产品级一般至少要区分：

- 短时记忆：最近几轮对话
- 会话记忆：同一会话内的阶段性总结
- 长期记忆：跨会话的用户偏好、已确认事实、稳定上下文

目前项目更像是“保存了历史消息”，但还没有把历史消息治理成可稳定复用的记忆系统。

### 3.3 上下文治理能力偏弱

当前已有 `TopK`、去重、`contextBudget` 之类的基础控制，但还没有真正的上下文压缩与精炼机制。

产品级系统通常还需要：

- 对长会话进行摘要压缩
- 对检索结果做 query-focused 压缩
- 控制无效上下文进入 prompt
- 在有限 token 内最大化有效证据密度

否则随着会话变长、文档变复杂，回答质量和延迟都容易波动。

### 3.4 检索前增强仍不完整

虽然项目已经有 query rewrite 和混合检索主链，但在产品级标准下，检索前能力还不够完整。

通常还需要：

- query 分类
- 术语归一和别名映射
- 多查询召回策略
- metadata filter 的更强利用
- 针对不同问题类型选择不同检索策略

这部分能力直接决定召回率和稳定性，是 Advanced RAG 的重要组成部分。

### 3.5 检索后校验仍偏基础

当前已经有引用、拒答和基础 grounding 校验，这是很好的基础。

但产品级还常见这些能力：

- 答案与引用的一致性检查
- 低置信度答案降级
- 错引、漏引修复
- 对最终答案做结构化后处理
- 对幻觉风险进行更强约束

换句话说，现在有“能拒答”，但还没有形成更成熟的答案质量控制链。

### 3.6 线上评测与反馈闭环不足

当前项目已经有离线评测系统，这是明显优点。

但产品级系统通常还需要进一步补齐：

- 线上真实问题采样与回流
- 用户反馈收集与标注
- 检索失败样本沉淀
- 版本对比、灰度发布、回滚策略
- 关键指标长期监控

只有离线评测还不够，必须形成线上数据驱动的迭代闭环。

### 3.7 观测性与运维体系不足

产品级不只是“能回答”，还要求“出了问题能看见、能定位、能回滚”。

通常需要补齐：

- 分阶段耗时监控
- 检索命中质量监控
- 失败率与拒答率监控
- 模型调用、重排服务、索引服务的可观测性
- 告警、限流、熔断、降级策略

如果这些能力不足，系统规模一上来就会进入“问题知道存在，但很难查清楚”的状态。

### 3.8 成本与性能治理不足

产品级 RAG 需要持续控制：

- 单次问答延迟
- embedding / 检索 / rerank / 生成成本
- 高并发时的资源使用
- 不同检索策略的成本收益比

当前链路已经具备较完整能力，但越完整的链路越容易带来成本和延迟压力，因此需要更系统的治理。

### 3.9 产品边界仍偏单一

当前项目的核心定位是“围绕单本书进行问答”，这与真正的通用知识产品仍有边界差异。

这并不是问题，但意味着：

- 当前系统更适合定义为“书籍问答 RAG 产品”
- 不应过早把它描述成通用 Agent 或全能知识助手
- 如果后续要扩展到多知识源、多工具、多任务，仍需补一层更强的路由与任务编排

## 4. 优先级建议

如果以“尽快向产品级靠近”为目标，建议优先级如下：

### 第一优先级

- 深化查询路由与策略开关化
- 短时记忆修正与会话摘要
- 上下文治理与压缩

这是最直接影响回答质量、稳定性和延迟的部分。

### 第二优先级

- 检索前增强补齐
- 检索后质量校验增强
- 线上评测与反馈回流

这是从“可用”走向“稳定可优化”的关键阶段。

### 第三优先级

- 可观测性、告警、降级、回滚
- 成本与性能治理
- 更复杂的数据源路由或工具能力

这是从“稳定系统”进一步走向“成熟产品”的基础设施层。

## 5. 总体判断

整体来看，OneBook AI 当前并不是“功能太少”，而是：

- RAG 主链已经成型
- 但产品级外围能力还没有完全补齐

因此更准确的定位应该是：

**当前项目已经具备中后期 Demo 到可用产品原型的能力，但距离成熟的产品级 RAG 系统，还缺少更强的路由治理、记忆、上下文治理、线上反馈闭环和运维观测等关键层。**

如果后续继续演进，建议优先把“问题是否该检索、历史如何压缩、上下文如何治理、效果如何在线闭环”这四个方向补上。它们会比继续单纯调检索参数更接近产品化价值。
