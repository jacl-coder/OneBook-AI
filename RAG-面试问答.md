# OneBook AI — RAG 面试问答（初稿）

> 用途：用于项目面试准备。该版本为初稿，你可逐条标注需要优化的回答。

## Q1. RAG 的工作机制是什么？
RAG 是“检索增强生成”：先从外部知识库检索与问题相关的证据，再把证据和问题一起交给大模型生成答案。它的核心价值是让回答有依据、可追溯，而不是只依赖模型参数中的记忆。

---

## Q2. RAG 有哪几个步骤？
标准流程可以拆成：
1. 数据准备（清洗、解析、分块、元数据）
2. 向量化与索引构建
3. 查询处理（规范化、改写、路由）
4. 检索与重排（召回 + 精排）
5. 证据约束生成
6. 评估与持续优化

---

## Q3. 选择的哪个向量数据库，为什么？
当前项目使用 Qdrant 作为向量检索存储，Postgres 只负责结构化数据和 chunk 元数据。这样检索路径与业务数据分层更清晰，也更符合当前 dense + sparse 混合检索的实现。

---

## Q4. 你怎样保证检索准确性？
我从四层保证准确性：
1. 数据层：清洗、结构化分块、元数据规范
2. 召回层：混合检索（dense + BM25）
3. 排序层：两阶段检索与重排、去重
4. 生成层：证据约束与引用一致性校验

---

## Q5. 如何提高召回率和准确性？
- 提召回：查询改写、多查询召回、提升初始 TopN
- 提准确：重排、元数据过滤、上下文去重打包、证据约束生成
实践策略是“先高召回，再高精排”。

---

## Q6. 你在实际项目中用了什么优化技巧？标准流程是什么？
我在问答链路里的优化重点是“先保证召回，再提升排序，最后约束生成”：

1. 查询侧优化：对用户问题做规范化、术语归一、查询改写；复杂问题做多查询召回，减少“问法差异”导致的漏召回。  
2. 召回侧优化：从单路向量检索升级为混合检索（dense + BM25），同时保留元数据过滤能力。  
3. 精排侧优化：使用两阶段检索（TopN 召回 + rerank 重排），并做去重与上下文打包，提升上下文有效密度。  
4. 生成侧优化：强制基于证据回答，要求引用；证据不足时拒答，降低幻觉。  
5. 后校验优化：对答案做 groundedness/引用一致性检查，不通过则降级或重试。  

标准流程是：先做高召回（Recall），再做高精度（Precision/nDCG），最后做真实性约束（Faithfulness）。

## Q7. 你的 RAG 怎么做效果评估？
我用“三层评估 + 双数据源”的方式做效果评估：

1. 检索层（Retriever）  
   - 指标：Recall@K、nDCG@K、MRR  
   - 目标：评估“能不能召回 + 排名是否合理”。

2. 生成层（Generator）  
   - 指标：答案正确率、引用命中率、幻觉率、拒答正确率  
   - 目标：评估“答得对不对、有没有证据、该拒答时是否拒答”。

3. 工程层（System）  
   - 指标：P95 延迟、失败率、单次问答成本  
   - 目标：评估“线上是否可用、可控、可持续”。

数据上我会同时看：  
- 固定离线评测集：做版本对比，保证可复现。  
- 线上抽样回流集：覆盖真实用户问法与边界 case。  

核心原则是：离线指标提升但线上指标恶化，不算优化成功。

## Q8. 你认为 RAG 为什么还会出现幻觉？怎么解决？
RAG 仍会幻觉，核心不是“模型太差”，而是“证据链断了”。常见断点有四类：

1. 没召回到关键证据  
   - 原因：问题表达与语料表达不一致、召回策略单一。  
   - 方案：查询改写 + 多查询召回 + 混合检索。

2. 召回到了，但排序不对  
   - 原因：噪声 chunk 排在前面，关键证据被淹没。  
   - 方案：两阶段检索（TopN + rerank）+ 去重打包。

3. 证据够，但生成没被约束  
   - 原因：提示词允许模型自由补全。  
   - 方案：强制证据约束输出、必须引用来源。

4. 有答案但无证据校验  
   - 原因：缺少输出后检查。  
   - 方案：groundedness / 引用一致性校验，不通过就拒答或重试。

一句话总结：先修检索，再加约束，最后加校验，三层一起做才能真正降幻觉。

## Q9. 边界 case 你怎么解决？
我一般先把边界 case 分成“输入问题、检索问题、状态问题、输出问题”四类，再分别兜底：

1. 输入问题：扫描版 PDF、低质量文本、格式噪声  
   - 做法：先走 native 提取，再按页质量触发 OCR；对文本做清洗和标准化。  
   - 目的：减少“源数据不可读”导致的后续误检索。

2. 检索问题：用户问法很短、很口语、和语料措辞差异大  
   - 做法：query normalize + 查询改写 + 多查询召回；再用重排去噪。  
   - 目的：减少漏召回和错召回。

3. 状态问题：书籍还在 queued/processing，用户已发起问答  
   - 做法：在服务层做状态门禁，未 ready 直接返回可解释提示，不进入生成链路。  
   - 目的：避免“无索引情况下强行回答”。

4. 输出问题：有回答但缺证据，或证据不足  
   - 做法：强制引用；证据不足时标准化拒答；必要时触发降级策略。  
   - 目的：控制幻觉并保证可追溯性。

一句话：边界 case 不是补丁处理，而是按链路分层设置护栏。

## Q10. 项目落地有哪些痛点，你如何解决？
这个项目落地时我遇到的痛点主要有三类：

1. 数据质量不稳定（尤其 PDF）  
   - 痛点：同是 PDF，文本层质量差异很大，直接影响分块和检索。  
   - 解法：建立“native 提取 + OCR 回退 + 页级质量融合”策略，并把提取方式写入 metadata 做追溯。

2. 质量和性能的冲突  
   - 痛点：加改写、多查询、重排会提升质量，但也会拉高延迟和成本。  
   - 解法：采用两阶段检索和参数化预算（TopN、TopK、重排开关），用 P95 与成本做门禁。

3. 优化不可验证、上线风险高  
   - 痛点：很多优化“看起来有效”，但上线后不稳定。  
   - 解法：固定离线评测集 + 线上抽样回流；策略开关化；灰度发布与回滚预案。

所以我的落地思路是：先把数据护栏和证据链建稳，再做检索提准，最后用评测和门禁保证可持续迭代。

## Q12. 详细讲一下 Golang 中 Channel 的概念和作用，它是否是并发安全的？
Channel 本质上是 Go 提供的“协程之间通信”的管道，用来在多个 goroutine 之间安全地传递数据和同步执行节奏。它体现的是 Go 提倡的思路：不要通过共享内存来通信，而要通过通信来共享内存。

它的几个核心作用是：
1. 数据传递：一个 goroutine 生产数据，另一个 goroutine 消费数据。
2. 同步控制：通过阻塞发送或接收，让多个 goroutine 按预期时序执行。
3. 解耦协作：生产者和消费者不需要直接操作同一块共享数据。

从并发安全角度讲，Channel 本身是并发安全的。多个 goroutine 同时对同一个 channel 做发送和接收，运行时会负责同步，不需要额外加锁。但“通过 channel 传递的数据”如果本身还会被多个 goroutine 共享修改，那数据对象自身仍然需要额外的并发保护。

---

## Q13. Channel 和传统的锁（Mutex）在实现并发控制时有什么区别？各自的适用场景是什么？
两者都能解决并发问题，但思路完全不同：

1. Mutex 解决的是“共享资源互斥访问”  
   - 核心是保护一段临界区，同一时刻只允许一个 goroutine 进入。  
   - 适合场景：多个 goroutine 需要读写同一个 map、计数器、状态对象。

2. Channel 解决的是“任务协作和数据流转”  
   - 核心是用消息传递来组织并发，而不是让大家直接抢同一份数据。  
   - 适合场景：生产者消费者、任务分发、worker pool、事件通知、pipeline。

简单说：
- 如果重点是“保护共享状态”，优先考虑 Mutex。
- 如果重点是“组织 goroutine 之间的协作关系”，优先考虑 Channel。

工程上我通常不会把它们对立起来，而是按职责选型：状态保护用锁，任务编排用 channel。

## Q14. 讲一下 GMP 模型。
GMP 是 Go 运行时的调度模型：

1. G（Goroutine）  
   - 代表一个协程，是用户态的轻量执行单元。

2. M（Machine）  
   - 代表真正执行代码的系统线程。

3. P（Processor）  
   - 代表调度上下文，可以理解为运行 goroutine 所需的逻辑处理器，里面包含本地运行队列等资源。

G 不能直接跑，必须绑定到 M 上执行；M 想执行 Go 代码，又必须持有 P。最终是 `P 把可运行的 G 交给 M 执行`。

GMP 模型的价值在于：
- 把大量 goroutine 复用到少量线程上，降低线程切换成本。
- 通过 P 的本地队列减少全局锁竞争。
- 借助 work stealing、syscall 切换、网络轮询等机制提高整体调度效率。

---

## Q15. 当 P 的本地队列为空或者不为空时，它会怎么去调度 G（协程）？
可以分两种情况看：

1. P 的本地队列不为空  
   - P 会优先从自己的本地队列取出可运行的 G，交给当前绑定的 M 执行。  
   - 这样做的好处是局部性更好、调度开销更低，也能减少对全局队列的竞争。

2. P 的本地队列为空  
   - 它会先尝试从全局队列拿一批 G。  
   - 如果全局队列也没有，就会去其他 P 那里做 work stealing，偷一部分可运行的 G。  
   - 再没有的话，会检查 netpoll 中是否有因为网络 IO 就绪而可恢复运行的 G。  
   - 如果仍然没有可运行任务，M 可能会进入休眠，等待新的调度机会。

所以 Go 调度的核心策略是：先本地，再全局，再偷取，最后看 IO 事件，尽量让线程不空转。

## Q16. Redis 支持哪些数据结构？
Redis 常见数据结构包括：

1. String：最基础，适合缓存对象、计数器、分布式锁标记。
2. List：适合消息队列、时间线、简单顺序集合。
3. Hash：适合存用户信息、商品属性这类对象字段。
4. Set：适合去重、标签集合、共同好友这类场景。
5. Sorted Set（ZSet）：带分值排序，适合排行榜、延时队列、范围查询。
6. Bitmap：适合签到、用户活跃标记、布尔状态压缩存储。
7. HyperLogLog：适合做 UV 这类基数统计。
8. Geo：适合地理位置检索，比如附近的人、门店距离。
9. Stream：适合消息流、消费组、可回溯消费场景。

面试里我一般会补一句：业务里最常用的还是 String、Hash、Set、ZSet。

## Q17. 为什么 Redis 的速度这么快？
Redis 快，核心不是单一原因，而是几件事叠加：

1. 基于内存  
   - 数据主要在内存里，避免了磁盘随机 IO。

2. 单线程处理命令  
   - 核心命令执行线程是单线程，避免了大量锁竞争和线程切换。

3. 数据结构高效  
   - Redis 针对不同场景设计了高效的数据结构和编码方式。

4. 网络模型高效  
   - 基于 IO 多路复用处理大量连接，避免一个连接一个线程的高开销模式。

5. 操作通常很短  
   - 大多数命令时间复杂度低，执行路径短。

需要补充的是：Redis 6 以后网络读写引入了多线程优化，但命令执行主路径依然以单线程为核心，所以“快”的本质仍然是内存 + 简洁模型 + 低开销。

## Q18. 如何实现一个类似淘宝搜索框的实时商品名称模糊搜索功能？
我会把这个需求拆成“输入联想”和“正式搜索”两层来做。

实时联想层：
1. 前端对输入做 debounce，比如 100 到 300ms，避免每敲一个字都发请求。
2. 服务端接收关键词后做前缀匹配、拼音匹配、模糊召回或热词补全。
3. 结果通常返回 TopN 个候选词，而不是完整商品列表。
4. 对高频词做 Redis 缓存，降低搜索引擎压力。

正式搜索层：
1. 用户点击搜索后，再走完整检索链路。
2. 一般会进入 Elasticsearch / OpenSearch 这类搜索引擎，支持倒排索引、分词、相关性打分、过滤、排序、分页。
3. 商品名称模糊搜索通常会结合分词、同义词词典、拼音、品牌词、类目词做综合召回。

如果让我做工程落地，我会优先选 `前端防抖 + 搜索服务 + ES 倒排索引 + Redis 热词缓存` 这条路线。

## Q19. 实时输入联想与输入完成后点击搜索在技术实现上有什么本质区别？
本质区别不在“都是查数据”，而在目标和链路设计不同：

1. 实时输入联想  
   - 目标是低延迟反馈，帮助用户补全意图。  
   - 返回的是“候选词”或“建议词”。  
   - 更关注响应速度、请求频率控制、缓存命中率。

2. 点击搜索  
   - 目标是给出完整、可排序、可筛选的正式结果。  
   - 返回的是“商品集合”。  
   - 更关注召回率、相关性排序、分页、筛选、统计聚合。

所以前者更像 suggestion service，后者更像 full search service。两者底层都可能依赖搜索引擎，但索引设计、返回结构、性能目标都不一样。

## Q20. 实时搜索通常使用什么网络协议（如 WebSocket）？你了解或有使用过吗？讲一下。
实时输入联想不一定非要 WebSocket。大多数搜索联想场景，用普通 HTTP 请求加前端 debounce 就已经足够，因为它本质上是“客户端频繁发起短请求，服务端快速返回建议”。

WebSocket 更适合：
1. 服务端需要主动推送数据。
2. 双向实时交互频繁。
3. 连接建立成本高，想复用长连接。

比如在线协同、IM、实时行情、流式 AI 输出，这些场景更适合 WebSocket。

如果是搜索框联想，我会优先选 HTTP / HTTPS；如果是“搜索结果持续刷新、服务端主动推送推荐、或者要承载更强实时交互”，才会考虑 WebSocket。它的核心特点是一次握手后保持长连接，客户端和服务端都可以主动发消息，延迟低，但连接管理、心跳保活、断线重连、网关扩展也更复杂。

## Q21. 请详细说明微信扫码登录的完整流程和背后发生的原理。
扫码登录本质上是“PC 端登录态获取”委托给“已登录微信的手机端”来完成身份确认。完整流程通常是：

1. PC 端请求登录二维码  
   - 服务端生成一个临时登录凭证，比如 `login_token`，并把它编码进二维码。  
   - 同时在服务端保存这个 token 的状态，初始为“待扫码”。

2. 用户用手机微信扫码  
   - 手机端识别二维码里的临时 token，并把它上报给微信服务端。  
   - 微信确认“是谁扫了这个码”。

3. 手机端确认登录  
   - 用户在手机上点确认，相当于用已登录的微信账号对本次 PC 登录授权。  
   - 这一步通常会校验账号身份、设备信息、二维码是否过期。

4. 服务端更新登录状态  
   - 服务端把这个 `login_token` 的状态从“待扫码”改成“已确认”，并绑定对应用户身份。

5. PC 端轮询或长连接感知状态变化  
   - PC 端不断轮询，或者通过 WebSocket / SSE 等方式收到“已确认登录”事件。  
   - 一旦确认成功，服务端给 PC 端签发 session 或 JWT，建立登录态。

它背后的关键原理有三个：
1. 二维码里放的是一次性临时凭证，不是用户真实身份。
2. 真正完成身份认证的是手机端，因为手机微信本身已经有登录态。
3. PC 端拿到的不是“扫码结果”，而是“服务端确认后的会话令牌”。

## Q22. 在微服务架构中，服务发现和负载均衡是如何实现的？
微服务里，实例地址是动态变化的，所以通常不会把 IP 写死，而是通过“注册中心 + 负载均衡”实现调用。

1. 服务发现  
   - 服务启动后，把自己的地址、端口、元数据注册到注册中心。  
   - 调用方从注册中心获取目标服务的可用实例列表。

2. 负载均衡  
   - 在多个可用实例之间选择一个发起请求。  
   - 常见策略有轮询、随机、最少连接、加权轮询、一致性哈希。

实现方式一般有两类：
1. 客户端负载均衡  
   - 调用方自己从注册中心拉实例列表，并在本地做选址。  
   - 典型思路类似 Ribbon / Spring Cloud LoadBalancer。

2. 服务端负载均衡  
   - 调用方只访问一个统一入口，由网关、Nginx、Envoy、SLB 负责转发。  
   - 典型场景是 API Gateway 或 Service Mesh。

---

## Q23. 服务注册中心（如 Nacos、Consul）是如何工作的？服务实例如何注册和保活（如通过心跳机制）？
注册中心本质上是“服务实例元数据的集中管理系统”。它一般维护一张服务目录表，记录某个服务名下有哪些实例、实例状态如何、是否健康。

典型流程是：

1. 服务启动时注册  
   - 实例把自己的 `serviceName`、IP、端口、版本、机房等信息注册到注册中心。

2. 注册中心保存并对外提供查询  
   - 消费方按服务名拉取实例列表，通常会本地缓存，定期更新。

3. 实例定期保活  
   - 服务实例会周期性发送心跳。  
   - 如果在超时时间内没收到心跳，注册中心会把它标记为不健康，或者直接摘除。

4. 健康检查与变更通知  
   - 有些注册中心除了心跳，还支持主动健康检查。  
   - 当实例上下线时，可以通过长轮询、推送或 watch 机制通知客户端更新。

所以它解决的是两个问题：让调用方“知道去哪找服务”，以及“知道哪些实例还能用”。

## Q24. 讲一下 Agent 中的“长短期记忆”。
Agent 的记忆通常分成短期记忆和长期记忆，两者作用不同：

1. 短期记忆  
   - 是当前任务或当前对话窗口内临时保留的信息。  
   - 比如最近几轮对话、当前目标、当前工具执行结果、中间推理状态。

2. 长期记忆  
   - 是跨会话、跨任务仍然有价值的信息。  
   - 比如用户偏好、常用设置、长期事实、业务规则、已沉淀的经验知识。

短期记忆更强调“当前上下文连续性”，长期记忆更强调“历史可复用性”。一个好的 Agent 往往是短期记忆负责当前推理，长期记忆负责跨轮复用。

---

## Q25. 什么样的信息应该放在长期记忆，什么样的信息放在短期记忆？
区分标准主要看两个维度：是否稳定、是否跨任务复用。

适合放长期记忆的：
1. 用户稳定偏好，比如语言习惯、展示格式、常用配置。
2. 长期身份信息，比如角色、团队、权限边界。
3. 可复用知识，比如业务规则、FAQ、项目背景、已确认事实。
4. 多次交互中反复出现、且后续仍可能有用的信息。

适合放短期记忆的：
1. 当前任务目标和步骤状态。
2. 最近几轮对话的上下文。
3. 本轮工具调用结果、中间草稿、临时约束。
4. 很快就会过期的即时信息。

一句话：稳定可复用的放长期，临时为当前任务服务的放短期。

## Q26. 当对话轮数很多，上下文窗口不足时，有哪些处理策略？
常见策略有四类：

1. 截断  
   - 只保留最近几轮最相关的上下文。  
   - 优点是简单直接，缺点是容易丢长期信息。

2. 摘要压缩  
   - 把前面对话总结成结构化摘要，再替代原始长上下文。  
   - 适合长对话持续进行的场景。

3. 检索式记忆  
   - 不把所有历史都塞进上下文，而是在需要时从记忆库中检索相关片段。  
   - 这是更可扩展的做法。

4. 分层上下文  
   - 把系统指令、当前任务、近期对话、长期记忆分层管理，只把最必要的内容拼接进 prompt。

工程上我更倾向“近期上下文 + 摘要 + 检索式记忆”的组合，而不是单纯不断堆 token。

---

## Q27. 如果要进行记忆压缩，通常有哪些方法？
常见方法包括：

1. 摘要压缩  
   - 把多轮对话总结成一段更短的自然语言摘要。

2. 结构化压缩  
   - 把对话提炼成字段，比如“用户偏好、目标、限制条件、待办事项”。

3. 关键信息抽取  
   - 只保留事实、约束、决策结果，去掉寒暄和冗余表达。

4. 分层存储  
   - 高频使用的信息保留详细版本，低频信息只保留摘要或索引。

5. 向量化检索  
   - 压缩后的记忆不直接全量注入，而是建立索引，在需要时再召回。

如果从工程可用性看，我会优先采用“结构化摘要 + 检索召回”，因为它比纯自然语言摘要更稳定，也更容易控制。

## Q28. 了解过 Agent 的设计范式吗？有哪些？
常见 Agent 设计范式包括：

1. ReAct  
   - 推理和行动交替进行，边想边调用工具。

2. Plan-and-Execute  
   - 先规划任务，再分步骤执行，适合复杂长任务。

3. Tool Use / Function Calling  
   - 模型根据输入选择合适工具，偏工程稳定性。

4. Multi-Agent  
   - 把不同角色拆成多个 Agent 协作，比如规划、执行、评审分开。

5. Reflection / Self-Refine  
   - 先生成结果，再让模型自查、自评估、自修正。

6. Memory-Augmented Agent  
   - 引入长短期记忆机制，提升跨轮连续性和个性化能力。

我理解这些范式不是互斥关系，实际系统里经常是组合使用，比如 `ReAct + Tool Calling + Memory + Reflection`。

## Q29. 你设计的 Agent 是怎么实现 ReAct 模式的？详细讲讲。
如果我用工程视角讲 ReAct，我会把它拆成一个“推理 - 决策 - 执行 - 观察 - 再推理”的闭环：

1. 输入理解  
   - 先解析用户目标、约束条件、是否需要外部工具。

2. Thought  
   - 模型先生成当前判断：问题是否明确、下一步要做什么、缺什么信息。  
   - 这一步不一定对外展示，但在内部作为决策依据。

3. Action  
   - 如果需要外部能力，就调用对应工具，比如搜索、数据库查询、代码执行、知识库检索。

4. Observation  
   - 把工具返回结果重新喂给模型，作为新证据。

5. 再决策  
   - 模型根据 observation 判断：是否已经足够回答，还是继续下一轮工具调用。

6. Final Answer  
   - 当证据足够时输出最终答案；如果证据不足，就明确说明不确定性或继续请求补充信息。

如果落到系统设计上，我会加三层控制：
1. 工具路由层：限制模型只能调用白名单工具。
2. 状态管理层：记录每一步的 thought、action、observation 和任务状态。
3. 安全治理层：限制最大步数、超时、重试次数、敏感工具权限。

所以 ReAct 的核心不是“让模型自己想”，而是让模型在可控框架里，基于外部观察不断修正行动路径。这比一次性直接回答更适合复杂任务。
